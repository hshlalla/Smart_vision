# 최종 비디오 대본 (한국어, 페이지별 설명 버전)

이 문서는 **UI를 직접 녹화하면서 페이지와 버튼 기준으로 설명하는 최종 비디오 대본**이다.  
로그인 페이지부터 시작해서, 검색, 인덱싱, catalog, agent까지 실제 화면 순서대로 읽을 수 있게 구성했다.

## 1. 영상 구성 원칙

이 비디오는 다음 기준으로 촬영하는 것이 좋다.

- 로그인 페이지부터 실제 웹 흐름 순서대로 보여준다.
- 각 페이지에서 **무슨 목적의 페이지인지** 먼저 설명한다.
- 핵심 버튼은 **누르기 전 설명, 누른 뒤 결과 설명** 순서로 말한다.
- 너무 세부 구현 설명으로 가지 말고, 사용자가 무엇을 할 수 있는지 중심으로 말한다.
- 마지막에는 시스템의 강점과 현재 한계를 짧게 정리한다.

---

## 2. 페이지별로 강조하면 좋은 핵심

### 로그인 페이지
- 이 시스템이 웹 기반으로 동작하며, 사용자 흐름이 분리돼 있다는 점
- 단순 모델 데모가 아니라 실제 서비스처럼 구성되어 있다는 점

### 검색 페이지
- 텍스트와 이미지를 함께 사용한 hybrid retrieval
- shortlist 기반 결과
- evidence-backed ranking

### 인덱싱 페이지
- 이미지 업로드 후 metadata preview
- 사용자가 수정 후 confirm하는 human-in-the-loop 구조
- 여러 장 이미지 사용 가능

### Catalog 페이지
- PDF를 searchable knowledge source로 바꾸는 기능
- 내부 문서 근거를 검색할 수 있다는 점

### Agent 페이지
- hybrid search, catalog search, 외부 근거를 조합할 수 있는 orchestration
- 단순 검색을 넘는 decision-support 성격

### 언어 전환
- bilingual UI 제공
- 실제 사용자 접근성 향상

## 3. 최종 대본

안녕하세요. 제 프로젝트는 **Smart Image Part Identifier for Secondhand Platforms**입니다.  
이 프로젝트는 중고 플랫폼에서 부품을 등록하거나 식별할 때, 사용자가 업로드한 이미지를 바탕으로 관련 후보를 찾고, 메타데이터를 보완할 수 있도록 돕는 웹 기반 시스템입니다.

먼저 로그인 페이지부터 보겠습니다.  
이 페이지는 사용자가 시스템에 들어오는 시작점입니다.  
이 프로젝트는 단순히 개별 모델을 실행하는 데모가 아니라, 실제 사용자 흐름을 가진 웹 애플리케이션으로 구성되어 있습니다.  웹 애플리케이션에는 언어 전환 기능도 포함되어 있습니다.  
사용자는 한국어와 영어 인터페이스를 전환할 수 있습니다.  
이 부분은 핵심 알고리즘과 직접 관련된 기능은 아니지만, 실제 사용성과 제출 완성도를 높이는 데 중요한 요소입니다. 저는 한국인이기 때문에 한국어를 포함해서 사용할수 있도록 개발하였습니다. 버튼으로 전환이 가능합니다.
로그인 후에는 검색, 인덱싱, catalog, agent 기능으로 이동할 수 있습니다.

다음은 인덱싱 페이지입니다.  
이 페이지는 새로운 아이템을 등록할 때 사용하는 화면입니다.  
사용자는 한 아이템의 이미지 한 장 또는 여러 장을 업로드할 수 있습니다. 아이템의 다양한 부분을 업로드 해주면 검색에서 더 수월하겠죠?
그리고 메타데이터를 처음부터 전부 직접 입력하는 대신, vlm 모델이 먼저 이미지를 보고 초안을 만들어 줍니다.
버튼을 눌러보겠습니다.

여기서 메타 자동 추출을 실행하면, 시스템은 qwen, 혹은 GPT 기반으로 maker, category, part number, description 같은 metadata preview를 생성합니다.  
사용자는 이 초안을 그대로 저장하는 것이 아니라, 내용을 확인하고 필요하면 수정할 수 있습니다.  
그리고 마지막으로 **confirm**을 눌렀을 때만 실제 인덱싱과 저장이 이루어집니다.
이 구조는 완전 자동 저장보다 더 안전하고, 실제 중고 등록 workflow에 더 잘 맞습니다.  
또한 여러 장의 이미지를 함께 활용할 수 있기 때문에, 앞면, 옆면, 라벨면처럼 서로 다른 시점의 정보를 함께 반영할 수 있습니다.  
이 부분은 fine-grained part identification에서 특히 중요합니다.
OCR 기능을 이용하여 할루시네이션을 방지할수도 있습니다. 텍스트에서는 ocr이 더 많은 강점을 가지기 때문에 OCR 버튼을 이용해 label을 스캔하는것도 좋은 방법입니다. 

이제 검색 페이지를 보겠습니다.  
이 페이지는 사용자가 텍스트만 입력하거나, 이미지와 텍스트를 함께 넣어서 관련 부품을 찾는 공간입니다.  
여기서 중요한 점은 이 시스템이 단순 분류기가 아니라 **retrieval-first 방식**으로 동작한다는 것입니다.  
즉 하나의 정답을 강제로 예측하기보다, 관련 후보를 shortlist로 보여주고 사용자가 판단할 수 있게 합니다. 부품은 연관성 있는 부품의 검색이 중요하기 때문에 다음과 같은 기능이 매우 유용할것입니다.

여기에서 텍스트를 입력할 수도 있고, 이미지를 업로드할 수도 있습니다.  
검색 버튼을 누르면 시스템은 이미지 임베딩, 텍스트 정보, 메타데이터 신호, lexical matching을 함께 사용해서 후보를 찾습니다.  
그래서 결과는 단순히 “비슷한 이미지” 순서가 아니라, maker나 part number 같은 증거도 함께 반영할 수 있습니다.
버튼을 눌러보겠습니다.

검색 결과를 보면, 이 시스템은 단순히 순위만 보여주는 것이 아니라 **evidence-backed 결과**를 제공하려고 설계되어 있습니다.  
즉 사용자는 결과를 그대로 믿는 것이 아니라, 왜 이 결과가 나왔는지 보고 판단할 수 있습니다.  
이것이 이 프로젝트의 중요한 강점 중 하나입니다.


다음은 Catalog 페이지입니다.  
이 기능은 PDF 문서를 업로드하고, 그 안의 내용을 검색 가능한 형태로 만드는 역할을 합니다.  
예를 들어 부품 카탈로그나 매뉴얼 같은 문서를 넣으면, 이후 검색 과정에서 이 문서들을 추가 근거로 활용할 수 있습니다.  
즉 시스템은 이미지 검색만 하는 것이 아니라, 사용자가 가진 reference document도 함께 검색 가능한 지식원으로 활용할 수 있습니다.

다음은 Agent 페이지입니다.  
이 기능은 hybrid search, catalog search, 그리고 필요할 경우 외부 검색을 조합해서 더 풍부한 답변을 제공하기 위한 인터페이스입니다.  
즉 단순히 후보를 보여주는 데서 끝나는 것이 아니라, 여러 근거 source를 묶어서 사용자의 판단을 지원하는 방향으로 확장할 수 있도록 설계했습니다.
또한 등록이 되어 있지 않아 모르는 제품을 웹에서 검색하였을때 아래 milvus에 바로 업데이트 가능하도록 upsert 기능을 추가하였습니다.

정리하면, 이 프로젝트의 핵심 강점은 단일 모델 성능 자체보다도, **검색, 메타데이터 보조, catalog 검색, agent orchestration, 그리고 사용자 확인 흐름을 하나의 실제 workflow로 통합했다는 점**입니다.  
즉 이 시스템은 단순 이미지 분류기가 아니라, **retrieval-first, human-in-the-loop identification assistant prototype**으로 보는 것이 가장 적절합니다.

현재 단계에서 이 시스템은 end-to-end로 동작하는 검색과 인덱싱 흐름을 갖추고 있고, retrieval, OCR, latency, usability를 포함한 실험과 평가도 완료하였습니다.  
다만 이 시스템의 성격상, 완전 자동 식별 시스템이라고 주장하기보다, 실제 사용 흐름을 지원하는 **retrieval-first, human-in-the-loop identification assistant**로 설명하는 것이 가장 정확합니다.

이상으로 제 프로젝트 소개를 마치겠습니다. 감사합니다.

촬영하면서 더 짧게 끊어 말하고 싶으면 아래 문장을 써도 된다.

### 로그인 페이지
- “이 페이지는 시스템의 시작점입니다.”

- “이 페이지는 이미지와 텍스트를 함께 사용하는 hybrid retrieval 화면입니다.”

### Agent 페이지

## 6. 발표 시 가장 강하게 내세울 강점

1. **실제 웹 애플리케이션 형태로 동작한다는 점**
2. **retrieval-first shortlist 방식**
3. **evidence-backed 결과**
4. **preview -> edit -> confirm 인덱싱 구조**
5. **catalog와 agent까지 포함한 확장성**

## 7. 너무 길게 말하지 말아야 할 부분

비디오에서는 아래 내용을 길게 설명하지 않는 것이 좋다.

- 라이브러리 버전 충돌
- 로컬 머신 환경 문제
- 모든 실험 수치의 세부값
- 세부 코드 구조

이런 내용은 보고서에 남기고, 비디오에서는 **페이지 흐름과 기능 가치**를 중심으로 설명하는 것이 더 좋다.
